模块一：UI 响应式适配与美学升级
1. 现代化鸿蒙美学规范 (UI Aesthetics)
为了让前端更加专业、美观，彻底摒弃早期的粗糙感，需严格遵循 HarmonyOS 原生设计规范：

图标替换：全面废弃 Emoji，采用 ArkUI 系统图标（sys.media...）或统一的 SVG 线性图标，保持视觉语言一致。

微调圆角与阴影：摒弃过于夸张的大圆角（>16vp）和浓重阴影。卡片使用 6vp~8vp 圆角，背景色使用纯白（#FFFFFF）配合极浅的灰色页面底色（#F1F3F5）来区分层级。列表项之间使用细分隔线（0.5vp, #E5E5E5）。

色彩体系：主色调采用鸿蒙品牌蓝（如 #007DFF），成功状态使用沉稳绿（#34C759），异常/失败使用警示红（#FF3B30），文字层级分明（主标题 #182431，副标题 #989A9C）。

状态可视化：签到过程的5个阶段（权限申请 → 活体检测 → 图像质量评估 → 人脸比对 → 签到结果）不应仅用文字提示，应设计为一个横向的步骤条 (Stepper / Progress)，当前节点高亮，成功打勾，让用户清晰感知系统状态。

2. 多端响应式布局方案 (Responsive Strategy)
禁止全局锁定横屏，应用需根据设备的窗口宽度（WindowSize）自动推断布局形态（手机/折叠屏/平板）。

< 600vp (手机 / 竖屏极窄形态)

首页：单列流式布局。顶部（课程与班级选择） -> 中部（统计面板卡片） -> 下部（人员列表）。签到核心操作悬浮固定在底部（position: fixed 或 Stack 底部对齐）。

录入页：分步向导模式（Step-by-Step），第一步填信息，第二步录人脸。

记录页：列表展示（List），筛选条件收纳至顶部的下拉菜单（Select）或底部弹窗（ActionSheet）。

600vp ~ 839vp (折叠屏展开 / 平板竖屏 / 双栏形态)

全局：采用左右分栏布局（如 Navigation 组件的 Split 模式）。

录入页：左侧表单信息输入，右侧相机预览与人脸录入框，双线并行操作。

≥ 840vp (平板横屏 / 三栏形态)

首页：左侧（导航/班级课程树形筛选），中间（当前选中课程的人员列表与签到操作区），右侧（实时签到动态滚动列表）。

记录页：采用横向展开的报表/表格（DataPanel 或自定义 Grid），信息一览无余。

3. 特殊的屏幕旋转控制
正常页面跟随系统传感器自动旋转。

活体检测特殊处理：在唤起 @kit.VisionKit 活体检测前，记录当前屏幕方向，强制锁定为竖屏（部分活体算法强依赖竖屏摄像头参数）；活体检测结束（无论成功或取消），必须恢复为进入活体前的屏幕方向。

模块二：班级、课程与用户关系模型 (核心逻辑实现)
当前的单点设计（按全量用户统计）无法满足真实的教学场景。必须引入关系型数据库（RelationalStore / RDB）重构底层数据。

1. 实体关系模型 (ER Diagram) 映射
应用需构建以下五大核心表及两张关联表：

AdminClass (行政班级表)：学生归属的固定班级。

Course (课程表)：如“高等数学”、“鸿蒙开发”。

User (用户表/学生表)：存储学生姓名、学号、绑定的 ClassId、人脸特征文件路径等。

CourseEnrollment (选课关联表)：多对多关系。记录哪个学生选了哪门课。

AttendanceSession (考勤场次表)：极其关键！ 老师点击“发起签到”时，生成的一条记录。包含 CourseId、发起时间等。

SessionMember (应到名单快照表)：核心逻辑！ 在创建 AttendanceSession 的瞬间，系统查询 CourseEnrollment，把所有选了这门课的学生快照复制到此表中。

AttendanceRecord (签到记录表)：包含 SessionId, UserId, 签到状态(成功/迟到/缺勤), 签到时间。

2. 核心业务痛点实现思路 (Coding Agent 必读)
A. 解决“转班导致历史数据错乱”的问题（快照机制）
业务规则：历史的考勤记录必须固化。如果张三今天转班了，他昨天的缺勤记录依然要挂在昨天的班级课程下。
代码实现思路：
统计出勤率时，绝对不能使用 Count(User表)。而是使用 SessionMember（应到） 和 AttendanceRecord（实到） 进行比对计算。SessionMember 在考勤发起的瞬间就已经固化了当时的人员名单，后续 User 表如何变动均不影响历史报表。

B. 解决“防误删”的软删除机制 (Soft Delete)
业务规则：班级和课程一旦产生过历史签到数据，禁止物理删除，只能“停用/归档”。
代码实现思路：
在所有基础数据表（Class, Course, User）中加入 status 字段（1=活跃，0=停用）。
前端 UI 下拉列表查询时，附加条件 WHERE status = 1。管理员在后台查历史报表时，仍然可以通过外键关联到停用状态的课程和人员。

模块三：迭代与实施路线图 (Roadmap)
为了保证项目稳定过渡，建议 Coding Agent 按照以下顺序逐步重构：

Step 1: 算法与基础体验修复 (Hotfix)
修复“活体检测完成后再调用人脸比对”的时序逻辑与图像方向问题（确保 PixelMap 旋转角度正确送入比对引擎）。

优化比对阈值策略（例如相似度 > 85% 即算通过，需在后台可配置）。

实现活体检测页面的“进前锁竖屏，出后复原态”逻辑（调用 @kit.Window 模块的 setPreferredOrientation）。

Step 2: 数据库重构与数据清洗 (Data Layer)
引入 @kit.RelationalStore。

编写 SQL 初始化脚本，建立完整的 Class -> Course -> Session -> Record 关系网。

封装 DAO (Data Access Object) 层，提供面向对象的 API 操作（如 createSession(), recordAttendance()）。

Step 3: UI 响应式与美学重构 (View Layer)
基于 ArkUI 的 GridRow 和 GridCol 或者 mediaquery，实现全局断点响应式布局。

重构组件，替换低质图标，引入步骤条（Stepper）组件展示签到 5 阶段。

将旧版耦合在 UI 里的业务逻辑抽离到 ViewModel 中。

Step 4: 报表与周边功能完善 (Features)
基于新的数据关系，开发多维度统计报表（按课程查出勤率、按班级查缺勤名单）。

增加本地数据导出功能（生成 CSV 或通过文件分享接口发出）。

增加全局异常拦截与隐私数据清理机制（提供“清除所有人脸与考勤数据”的危险操作入口）。
